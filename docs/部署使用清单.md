## 阶段 0：生成 JWT Secret

Gateway 与 Document Server 共享同一个 JWT Secret，用于：
- Document Server 签发/验证编辑会话 token
- Gateway Admin 登录 session token
- Webhook HMAC 签名

### 生成方式

```bash
# 推荐：OpenSSL 随机生成 32 字节 hex
openssl rand -hex 32
# 输出示例: a1b2c3d4e5f6...（64 位 hex）

# 或者用 Python
python3 -c "import secrets; print(secrets.token_hex(32))"

# 或者用 Node.js
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

**生产环境务必自定义此值**。Docker Compose 一键部署时，将此值填入 `.env` 的 `JWT_SECRET`，会自动同步到 Document Server 和 Gateway 两个容器。

## 阶段 1：Docker Compose 一键部署（推荐）

### 1.1 配置

```bash
cp .env.example .env
```

编辑 `.env`，最少设置：

```env
JWT_SECRET=a1b2c3d4...     # 阶段 0 生成的值
ADMIN_PASSWORD=强密码        # 管理端登录密码
```

完整配置项见 `.env.example`，均有合理默认值。

### 1.2 启动

```bash
docker compose up -d
```

启动三个容器：

| 容器 | 端口 | 说明 |
|---|---|---|
| `oo-document-server` | 18000 | ONLYOFFICE Document Server |
| `oo-gateway` | 18080 | Gateway API |
| `oo-admin-ui` | 18081 | 管理端界面（nginx） |

### 1.3 访问管理端

浏览器打开 `http://<host>:18081/admin/login`，用 `.env` 中设置的 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 登录。

### 1.4 查看状态

```bash
docker compose ps
docker compose logs -f gateway
```

Gateway 访问日志格式：

```text
[http] GET /api/v1/health 200 320µs remote_addr=172.18.0.1:54321 user_agent="curl/8.0.1" request_id=-
```

所有业务 API、Document Server 内部 API、编辑器页面、健康检查和 Admin API 都会记录访问日志。排查请求链路时可透传 `X-Request-Id` 或 `X-Request-ID`，Gateway 会写入 `request_id` 字段。

## 阶段 2：手动部署（源码）

### 2.1 克隆项目

```bash
cd /opt
git clone <repo-url> onlyoffice-gateway
cd onlyoffice-gateway
```

### 2.2 配置

```bash
cp .env.example .env
```

编辑 `.env`：

```env
LISTEN_ADDR=0.0.0.0:18080
DOCUMENT_SERVER_URL=http://localhost:18000
JWT_SECRET=<阶段 0 生成的值>
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<强密码>
```

Gateway 启动时自动加载 `.env`，无需 `source` 或 `export`。

### 2.3 构建并启动

```bash
# 终端 1：启动 Gateway
make build
./bin/gateway

# 终端 2：启动管理端（开发模式）
make frontend-dev
```

管理端开发服务器运行在 `http://localhost:5173/admin/login`，通过 Vite proxy 自动将 `/admin/api` 转发到 Gateway 的 18080 端口。

### 2.4 单独部署 Document Server

如果不用 Docker Compose，手动启动 Document Server：

```bash
docker pull onlyoffice/documentserver:latest

docker run -d \
  --name onlyoffice-document-server \
  -p 18000:80 \
  -e JWT_SECRET=<与 Gateway 相同的值> \
  -v /opt/onlyoffice/DocumentServer/data:/var/www/onlyoffice/Data \
  onlyoffice/documentserver:latest
```

## 阶段 3：注册业务服务

打开管理端，登录后：

1. 点击 **Add Service**
2. 填写 **Service ID**（唯一标识，如 `crm-service`）
3. 粘贴业务服务的 **RSA 公钥**（PEM 格式）
4. 填写 **Webhook 域名白名单**（每行一个，或逗号/分号分隔）
5. 提交

新增的服务**立即生效**，无需重启 Gateway。后续可在管理端编辑或删除。

## 阶段 4：业务服务接入

### 4.1 生成 RSA 密钥对

```bash
openssl genpkey -algorithm RSA -out service-private.pem -pkeyopt rsa_keygen_bits:2048
openssl rsa -pubout -in service-private.pem -out service-public.pem
```

将 `service-public.pem` 的内容粘贴到管理端的 "RSA Public Key" 字段。私钥由业务服务保管，用于签名 JWT。

### 4.2 签 JWT 并上传文档

```javascript
const jwt = require('jsonwebtoken');
const fs = require('fs');
const FormData = require('form-data');

const GATEWAY_URL = 'http://your-gateway:18080';
const PRIVATE_KEY = fs.readFileSync('service-private.pem');

const token = jwt.sign({
  service_id:  'my-service',
  webhook_url: 'https://my-app.example.com/webhook',
  external_id: 'doc-001',
  file_name:   'document.docx',
  document_type: 'word',
}, PRIVATE_KEY, { algorithm: 'RS256', expiresIn: '60s' });

const form = new FormData();
form.append('file', fs.createReadStream('document.docx'));

const resp = await fetch(`${GATEWAY_URL}/api/v1/documents`, {
  method: 'POST',
  headers: { Authorization: `Bearer ${token}`, ...form.getHeaders() },
  body: form,
});
const { document_id } = await resp.json();
```

### 4.3 生成编辑 token

```javascript
app.get('/api/edit-token', (req, res) => {
  const token = jwt.sign({
    service_id:  'my-service',
    document_id: req.query.documentId,
  }, PRIVATE_KEY, { algorithm: 'RS256', expiresIn: '30m' });  // token 仅在浏览器首次加载 /edit 页面时校验一次，页面加载后不再使用
  res.json({ token });
});
```

### 4.4 业务 S3 直连（零字节 Gateway）

如果原文件已由业务服务存储在 S3，请不要上传 multipart 文件。将 Document Server 可访问的 HTTPS 预签名 GET URL 放入 RS256 JWT 的 `source_url`，以空请求体调用创建接口。Gateway 仅保存会话元数据，编辑完成 webhook 中的 `edited_url` 必须由业务服务立即下载并保存回自己的 S3；原文件和结果文件的生命周期均由业务 S3 管理。

`source_url` 必须是 HTTPS、不可包含用户名/密码，且有效期应覆盖编辑期间的读取。开发环境若使用 HTTP MinIO，请继续使用 multipart fallback，或通过受控 HTTPS 反向代理提供预签名 URL。直连模式下 Gateway 不会下载原文件或编辑后文件，也不会清理业务 S3 对象；业务服务收到 webhook 后应立即拉取 `edited_url` 并保存回自己的对象存储。

| 部署形态 | 推荐接入 | 需要确认 |
|---|---|---|
| 业务服务和 Document Server 都能访问公网 S3 | `source_url` 直连 | 预签名 GET URL 有效期覆盖编辑窗口，业务 webhook 能访问 `edited_url` |
| Document Server 在内网，S3 走内网域名或 VPC Endpoint | `source_url` 直连 | `source_url` 使用 Document Server 可解析、可访问的 HTTPS 域名 |
| 本地 MinIO 只有 HTTP 或自签证书 | multipart fallback | Gateway 本地 TTL 清理开启，或给 MinIO 配受信 HTTPS 反向代理 |
| 业务侧不能立即处理 webhook | multipart fallback | 由 Gateway 临时保存结果，业务稍后调用结果下载接口 |

业务 S3 直连模式建议：

- `source_url` 有效期建议不短于一次编辑会话的最大预期时长，短文档可从 2h 起步，长协作编辑应按业务会话续签或改用 multipart fallback。
- webhook 处理必须幂等：以 `document_id` 或 `external_id` 作为保存键，重复通知时覆盖同一版本或忽略已处理版本。
- 业务 S3 配置 Lifecycle：原始临时对象按业务规则过期；保存后的正式对象按业务数据策略保留。
- 不要依赖 Gateway 清理业务 S3；Gateway 只清理自己本地托管的 multipart 文件。

## 阶段 5：前端集成

### 5.1 安装 SDK

```bash
npm install @zenmind/onlyoffice-editor
```

### 5.2 嵌入编辑器

```tsx
import { OnlyOfficeEditor } from "@zenmind/onlyoffice-editor";

const { token } = await fetch(`/api/edit-token?documentId=${docId}`).then(r => r.json());

<OnlyOfficeEditor
  documentId={docId}
  gatewayUrl="http://gateway:18080"
  token={token}
  onSaved={async (e) => {
    const resp = await fetch(`${gatewayUrl}/api/v1/documents/${e.document_id}`, {
      headers: { Authorization: `Bearer ${await getServiceJWT()}` },
    });
    const blob = await resp.blob();
    await saveToStorage(blob);
  }}
  style={{ height: "80vh" }}
/>
```

## 常见问题

### Q: 如何生成 JWT_SECRET？

```bash
openssl rand -hex 32
```

Gateway 与 Document Server 必须使用相同的值。Docker Compose 部署时通过 `.env` 中的 `JWT_SECRET` 自动同步。

### Q: 管理端无法登录？

1. 确认 `.env` 中设置了 `ADMIN_PASSWORD`
2. 确认 Gateway 日志没有 `WARNING: ADMIN_PASSWORD not set`
3. Gateway 启动时会自动加载 `.env`，如果仍无密码，检查 `.env` 是否在正确目录

### Q: Document Server 无法访问 Gateway？

Gateway 需监听 `0.0.0.0`（容器默认配置），Docker Compose 已自动配置。

如果表现为打开文档慢或 Document Server 拉取文档慢，按顺序检查：

1. `/edit` 页面中生成的 `document.url` 是否是 Document Server 可达的 Gateway 地址
2. `GET /download/{docId}` 是否返回 `Content-Length`、`ETag`、`Accept-Ranges: bytes`
3. Gateway 与 Document Server 是否走容器内网或同 VPC 内网，避免服务间请求绕公网
4. Gateway 前置 Nginx/CDN 是否保留静态资源缓存头，特别是 `/web-apps/`、`/sdkjs/`、`/spellchecker/`、`/cache/`
5. `gateway-data` 是否部署在低延迟磁盘上，避免网络盘拖慢大文件读取

### Q: 如何启用 MinIO/S3 存储？

1. 设置 `STORAGE_BACKEND=s3`
2. 配置 `S3_ENDPOINT`、`S3_REGION`、`S3_BUCKET`、`S3_ACCESS_KEY`、`S3_SECRET_KEY`
3. MinIO 通常设置 `S3_USE_PATH_STYLE=true`、`S3_USE_SSL=false`
4. 确认 bucket 已创建；Docker Compose 本地开发可使用：

```bash
STORAGE_BACKEND=s3 docker compose --profile minio up -d
```

### Q: 上传时提示 "invalid token"？

检查：JWT 算法是否为 RS256、私钥与管理端注册的公钥是否配对、JWT 是否在有效期内。

### Q: Webhook 域名白名单检查失败？

检查的是 Webhook URL 的域名部分（不含 scheme 和端口）。在管理端编辑服务即可修改白名单。

### Q: 文档过期被清理？

TTL 默认 8 小时。编辑期间 callback（status=1）自动续期。可在 `.env` 中调整 `TTL_HOURS`。本地存储会按 `CLEANUP_INTERVAL`（默认 `1h`）自动扫描并删除过期文档；业务 S3 直连模式的对象生命周期由业务侧 S3 策略负责。

### Q: 如何观察保存和 webhook 状态？

访问 `GET /api/v1/metrics`。重点看：

- `onlyoffice_gateway_callback_save_queued_total`
- `onlyoffice_gateway_callback_save_dropped_total`
- `onlyoffice_gateway_callback_save_succeeded_total`
- `onlyoffice_gateway_callback_save_failed_total`
- `onlyoffice_gateway_webhook_succeeded_total`
- `onlyoffice_gateway_webhook_failed_total`

高并发保存时可调大 `.env` 中的 `CALLBACK_QUEUE_SIZE` 和 `CALLBACK_WORKERS`。如果 `callback_save_dropped_total` 增长，说明保存队列已经满，应扩容 Gateway 或降低回调峰值。

指标按 Prometheus text exposition 输出，包含 `HELP` 与 `TYPE`。建议告警：5 分钟内 `callback_save_dropped_total`、`callback_save_failed_total` 或 `webhook_failed_total` 增量大于零；告警内容同时带上 Gateway 实例标签（由 Prometheus 的 scrape target 标签提供），避免在 Gateway 指标中加入高基数的文档或服务 ID。

### Q: S3 Lifecycle 应如何配置？

直连 S3 模式中，Gateway 不保存对象字节，也不会删除业务对象。请在业务 bucket 上按前缀设置生命周期；下面是临时编辑源文件保留 7 天的 AWS S3 Lifecycle 示例，`Prefix` 必须替换为业务实际前缀：

```json
{
  "Rules": [{
    "ID": "expire-onlyoffice-temporary-sources",
    "Status": "Enabled",
    "Filter": { "Prefix": "onlyoffice/tmp/" },
    "Expiration": { "Days": 7 },
    "AbortIncompleteMultipartUpload": { "DaysAfterInitiation": 1 }
  }]
}
```

已保存的正式对象应由业务数据保留策略管理，不应复用这条临时对象规则。

### Q: 如何做上线前故障演练？

1. 用压测工具并发发送合法的 ONLYOFFICE 保存 callback，观察 `callback_save_queued_total` 增长；逐步提高并发直到出现 `callback_save_dropped_total`，据此设置队列、worker 数和 Gateway 实例数。
2. 让 Document Server 的编辑结果下载地址短暂返回 5xx，确认原文档不会被标记为已编辑，且 `callback_save_failed_total` 增长。
3. 让业务 webhook 连续返回 5xx，确认 Gateway 按指数退避加 jitter 重试、最终 `webhook_failed_total` 增长；业务 webhook 必须按 `document_id` 实现幂等。
4. 在有正在保存任务时向 Gateway 发送 `SIGTERM`，确认负载均衡停止引流后进程等待 HTTP 请求与已入队保存处理完成。容器的 `terminationGracePeriodSeconds` 应至少设为 35 秒。
